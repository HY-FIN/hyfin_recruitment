## 0. 계획 수립 원칙

1. **버그 먼저, 기능은 그다음.** 사용자가 기능을 못 쓰는 상태(로그인 불가, 점수 리셋)는 다른 어떤 개선보다 앞선다.
2. **데이터 스키마 변경은 한 번에 묶는다.** 지원서 항목 추가/변경은 Prisma 마이그레이션 + 폼 + 관리 시트 + 통계까지 파급되므로, 요구사항이 확정된 뒤 하나의 작업 단위로 처리한다.
3. **외부 의존(양식, 문항)이 있는 항목은 블로킹으로 분리**해 대기하지 말고, 나머지를 먼저 진행한다.
4. **리크루팅 일정이 실서비스 데드라인이다.** 면접 날짜가 19~21일로 언급된 것으로 보아 남은 기간이 약 1주. 이번 사이클에 반드시 필요한 것과 다음 사이클로 미뤄도 되는 것을 구분한다.
5. 이 프로젝트 원칙에 따라 **원인 분석·검증은 `/inspector`, 코드 수정은 `/implementor`** 서브에이전트에 위임한다.

## 3. 실행 순서 (스프린트 플랜)

리크루팅 일정(면접 19~21일)을 데드라인으로 역산한 순서.

### Day 1 — 버그 소탕
1. `/inspector`: P0 3건 원인 분석 (로그인, 점수 리셋, 개인질문 권한)
2. `/implementor`: 확정된 원인 순서대로 수정
3. 블로킹 항목 4건 관계자에게 요청 발송

### Day 2 — 지원서 양식 (마감 전 최우선)
지원서는 지원자가 실시간으로 제출 중일 수 있으므로 가장 먼저 확정해야 한다.
1. Prisma 스키마 변경: 군필/재·휴학, 학년(enum화), 관심분야 필드 추가 → 마이그레이션
2. **기존 제출 데이터 처리 방침 결정**: 새 필드는 nullable로 두고, 기존 지원자의 학년 자유입력 값은 통계 시 정규화 매핑
3. 지원 폼 + 관리 시트 상세 + 통계 화면 반영 (강이원 요구의 핵심은 통계 정확성)
4. "희망시간 → 가능시간" 문구 변경, 생년월일 드롭다운 기본값(P2-2) 끼워 처리

### Day 3 — 면접 슬롯 & 자동배치
1. 면접 날짜/시간 슬롯 설정 변경 (19~21일, 18:00~20:30)
2. 슬롯 정원 3명(5:3) 반영
3. 자동배치 로직 재작성: 1·2학년 / 3·4학년 그룹 → 그룹 내 전공 분산 순. 기존 배치 결과와의 호환성(이미 배치된 지원자 처리) 확인

### Day 4 — 관리 시트 & 메일
1. 합불 일괄 저장(로컬 상태 → 저장 버튼) + 최종합불 정렬 추가 + 미저장 이탈 경고
2. 메일 템플릿 교체 (양식 수령 완료 가정): 온보딩 회신 문구, 전화번호 포함
3. 서류 마감기간 설정 기능

### Day 5 — 검증 & 잔여
1. `verify` / QA: `docs/test/test_plan.md` 갱신 후 전체 플로우 회귀 테스트 (지원 → 서류 합불 → 메일 → 면접 배치 → 평가 → 최종 합불 → 메일)
2. 특히 **메일은 실제 발송 테스트** (테스트 계정으로 합/불 각 케이스)
3. P2-1 로딩 개선, 여유 시 P2-3 면접 편의 화면 착수
4. `/inspector` 보안 감사(P2-4) 결과 반영

## 4. 리스크 및 주의사항

- **스키마 마이그레이션**: 접수가 진행 중인 운영 DB(Supabase)에 마이그레이션을 적용해야 한다. 필드 추가는 nullable로만, 삭제/타입변경은 이번 사이클에서 금지.
- **점수 리셋 버그(P0-2)**: 이미 작성된 평가 데이터가 3점으로 덮어써졌을 가능성이 있다. 수정 전에 기존 평가 데이터 무결성을 먼저 확인할 것.
- **메일 발송**: 최근 OAuth2 전환(커밋 605e175) 직후이므로 템플릿 교체 시 발송 회귀 테스트 필수.
- **자동배치 재실행**: 슬롯 구조가 바뀐 뒤 기존 배치 결과가 무효화될 수 있음 — 재배치 시 안내 필요.
- **박지원 로그인**: 코드 문제가 아니라 시드 데이터/계정 등록 문제일 가능성이 높음. 코드부터 고치지 말고 데이터부터 볼 것.
